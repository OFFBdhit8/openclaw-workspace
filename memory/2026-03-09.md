- Reddit：被封（403），无法访问

**核心发现：**

1. **GPT 5.4 在 OpenClaw 中表现不佳的根本原因**
   - 工具调用可靠性差：不开 thinking mode，准确率明显下降
   - 自适应推理是劣势：有"快速路径"和"深度推理路径"，行为不一致
   - 指令遵循不如 Claude：thinking 必须 ≥ medium，否则工具调用不可靠
   - 过度探索：默认"thorough and comprehensive"收集上下文，导致无效工具调用

2. **社区共识**
   - Claude Sonnet 4.5：稳定、可预测、工具调用准确，适合长时间运行的 agent
   - GPT 5.4：推理强、灵活、但需要精细调参，适合"推理复杂度波动"的场景

3. **OpenAI 官方优化指南**
   - **reasoning_effort 参数是关键**：默认 medium，简单任务用 low/minimal，复杂任务用 high
   - **verbosity 参数控制输出长度**：全局设置 low，代码生成场景用自然语言覆盖为 high
   - **Tool Search**：减少 47% token 消耗（MCP Atlas 测试）
   - **Responses API**：比 Chat Completions 提升 4.3%（Tau-Bench 数据）
   - **指令冲突是性能杀手**：GPT 5.4 会花费大量推理 token 试图调和矛盾的指令

4. **Composio 实战数据**
   | 任务 | Claude Sonnet 4.5 | GPT 5 Codex |
   |---|---|---|
   | UI 克隆 | $10.26（18M tokens） | $2.50（600k tokens） |
   | 修复 lint 错误 | 失败（4M tokens） | 成功（100k tokens） |

**核心发现（社区实战）：**

### 1. GPT 5.4 的三大致命问题（bswen.com）

**问题 1：危险假设**
- 案例：建议 `DROP TABLE users` 删除生产数据库，没有确认
- 原因：选择最简单路径，不验证约束

**问题 2：技术幻觉**
- 编造不存在的 API 方法（如 `User.fetchById()` 实际是 `User.findById()`）
- 编造参数名、库函数

**问题 3：质量不一致**
- "有时很好，有时犯低级错误"
- 无法完全信任，也无法完全放弃

### 2. 272K Token 定价陷阱（apiyi.com）

**关键发现：**
- **272K 是硬性分界线**：超过后整个会话按双倍计费，不是只对超出部分
- **输出价格也涨**：$15→$22.50（涨 50%）
- **最佳性能区间**：127K-272K tokens（准确率 97%）
- **性能下降区**：256K 后准确率开始下降，512K-1M 可能降至 36%

### 3. GPT 5.4 vs GPT 5.3 Codex（nxcode.io）

| 场景 | 赢家 | 数据 |
|---|---|---|
| 通用编程 | 平局 | 都是 ~80% SWE-bench |
| 终端任务 | GPT 5.3 Codex | 77.3% vs 75.1% |
| 桌面自动化 | GPT 5.4 | 75% OSWorld（超人类基线） |
| Token 效率 | GPT 5.4 | 减少 47% tokens |

**关键洞察：**
- GPT 5.4 的 47% token 节省可以抵消更高的单价
- 272K 是成本悬崖：超过后价格翻倍

### 4. 已执行的新增优化

**1. 危险操作防护**
- 检测危险模式：DROP TABLE / DELETE FROM / rm -rf
- 暂停 → 说明风险 → 请求确认 → 记录日志
- 明确约束：NEVER suggest DROP TABLE without confirmation

**2. API 方法验证（防止幻觉）**
- 验证方法是否存在于文档中
- 验证参数名是否正确
- 不确定时明确说"I need to check the API documentation"
- 使用 Structured Outputs + Schema 验证

**3. Context 管理策略（避免 272K 陷阱）**
- 保持输入 <272K tokens（避免双倍计费）
- 重复内容放开头（利用 Prompt Caching，节省 90%）
- 大文档分段处理，每段 <272K
- 使用摘要压缩（先用便宜模型提取摘要）

**4. Chain-of-Thought 强制显示推理**
- 复杂计算必须显示推理步骤
- 格式：Step 1-5（输入 → 规则 → 计算 → 结果 → 验证）
- 推理透明，可审计
